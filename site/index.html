Votre agent code dans un navigateur qu'il ne voit pas. cdpx lui donne des yeux — et à vous, des preuves.

cdpx expose des primitives Chrome DevTools Protocol en ligne de commande. Famille 01 · Voir

Donner des sens à l'agent — et au dev qui le pilote

dev frontagentiquedev quotidien

Un agent qui ne lit ni la console JS ni les requêtes en échec debugge un front à l'aveugle. Console et réseau sont les deux boucles de feedback du dev front : ce sont les primitives les plus rentables de tout l'outil, complétées par la capture de pixels et la vision sémantique de l'arbre d'accessibilité.

consolenetworkmetricsscreenshotpdfa11y

Démo enregistrée — Chrome réel, site témoin du dépôt (source du cast) : console, réseau, métriques runtime.

« C'est corrigé » — sauf que la page est cassée · l'exception, la XHR en 500 et le clic mort deviennent trois JSON

L'agent annonce que c'est réglé ; vous découvrez le contraire en ouvrant le navigateur. Alors vous devenez copiste : capture d'écran collée dans le chat, stack retranscrite à la main, onglet Réseau décrit avec des mots.

l'agent constate lui-mêmeexemple des fiches
$ cdpx console --duration 3
{"entries":[{"kind":"console","type":"error","text":"TypeError: cart is undefined","ts":1751700000123.4}],"count":1,"errors":1,"duration":3.0}
$ cdpx network http://demo.test/checkout
{"url":"http://demo.test/checkout","requests":[{"url":"http://demo.test/api/cart","method":"GET","resourceType":"XHR","status":500,…}],"summary":{"total":14,"errors_4xx_5xx":1,"bytes":184320}}
$ cdpx dom-diff -- click "#offcanvas-cart"
{"action":["click","#offcanvas-cart"],"changed":false,"diff":[],"lines":0}

Ce que ça change : l'exception, la XHR en 500 et le clic mort sont trois JSON que l'agent lit, corrèle et re-vérifie après son fix. La boucle « corrige, puis prouve » ne passe plus par vos yeux.

Lire la page comme un lecteur d'écran · rôles et noms, sans le bruit du HTML

Le HTML complet coûte cher à lire — pour un agent comme pour un humain. L'arbre d'accessibilité est la vision sémantique de la page : rôles et noms, sans le bruit. Un bouton sans nom accessible y disparaît — c'est précisément le signal recherché.

la page en rôles et nomssortie réelle · site témoin
$ cdpx a11y
{"nodes":[{"role":"RootWebArea","name":"Fixture: formulaire","ignored":false},{"role":"heading","name":"Formulaire","ignored":false},…,{"role":"textbox","name":"Nom","ignored":false},…,{"role":"textbox","name":"Email","ignored":false},…,{"role":"button","name":"Envoyer","ignored":false},…]}

Ce que ça change : l'agent vérifie libellés, structure de titres et zones de repère sans parser le HTML — et les manques d'accessibilité deviennent visibles au même prix.

Famille 02 · Mesurer

Objectiver au lieu de constater

dev back Symfonydev frontperf

« C'est lent », « ça a l'air mieux » : tant que la performance est une impression, elle se discute. cdpx transforme le profiler Symfony, les Core Vitals et la couverture de code en nombres — et un nombre se compare, se suit, se refuse.

profilervitalscoverage

Démo enregistrée — vraie app Symfony témoin du dépôt (source du cast) : la même page en variante saine (3 requêtes, 0 doublon) puis N+1 (6 requêtes, 4 doublons, SQL à l'appui), et le one-liner jq -e qui refuse le merge.

Le N+1 Doctrine attrapé à la fiche produit, pas en prod · 6 requêtes dont 4 duplicats : un nombre, plus une intuition

Le WebProfiler a la réponse depuis le début, mais il vit dans un navigateur : personne ne clique neuf panels à chaque itération, et un agent ne peut pas le lire du tout.

le profiler devient une donnéeexemple des fiches
$ cdpx profiler http://127.0.0.1:8000/produit/42 --panels db,cache
{"token_present":true,"status":200,"panels":{"db":{"available":true,"queries":6,"statements":2,"duplicates":4,"time_ms":1.76,…},"cache":{"available":true,"calls":4,"hits":3,"misses":1,…}}}
$ cdpx profiler http://127.0.0.1:8000/produit/42 --panels db | jq -e '.panels.db.duplicates == 0' || echo "N+1 détecté"
N+1 détecté

Ce que ça change : 6 requêtes dont 4 duplicats — le N+1 est un nombre, plus une intuition. Le même one-liner, posé en CI, empêche le prochain (voir famille 06).

Famille 03 · Auditer

Auditer le rendu — celui que Googlebot indexe

product / SEOe-commercemigrations

Pour un audit SEO, ce qui compte est le DOM final vu par le rendering de Googlebot, pas le HTML servi. curl et grep s'arrêtent à la source ; cdpx extrait le contrat SEO du DOM rendu en un appel : title, metas, canonical, hreflang, JSON-LD, h1, alt, liens — plus des findings prêts à asserter.

seo

Démo enregistrée — Chrome réel, site témoin du dépôt (source du cast) : contrat SEO complet en un appel, findings vides sur la page conforme, nommés sur la page cassée.

Le contrat SEO, diffable entre staging et prod · le canonical qui disparaît se voit à la PR, pas dans Search Console

Les canonicals et hreflang cassent silencieusement pendant une refonte, et l'audit d'agence arrive après la chute de trafic.

le SEO devient un objet comparableexemple des fiches
$ cdpx seo https://staging.exemple.fr/produit-42 > staging.json
$ cdpx seo https://www.exemple.fr/produit-42 > prod.json
$ diff <(jq -S '{title,canonical,h1,robots}' staging.json) <(jq -S '{title,canonical,h1,robots}' prod.json)
<   "canonical": null,
>   "canonical": "https://www.exemple.fr/produit-42",
$ jq .findings staging.json
["2 image(s) sans alt"]

Ce que ça change : le contrat SEO est un JSON versionnable, comparé avant et après release — le canonical qui disparaît se voit à la PR, pas dans Search Console.

Auditer l'état après interaction, pas seulement l'URL · les états intermédiaires d'un parcours s'auditent comme des pages

Une SPA, une variante sélectionnée, un panier ouvert : l'état qui compte n'a parfois pas d'URL. cdpx seo sans argument audite la page actuellement affichée, quel que soit le chemin qui y a mené.

l'audit suit le parcoursexemple des fiches
$ cdpx goto http://shop.localhost/collection/robes
$ cdpx click "#variante-taille-38"
$ cdpx seo
# audite le DOM rendu de l'état courant — title, canonical, JSON-LD après le routage côté client

Ce que ça change : les états intermédiaires d'un parcours e-commerce s'auditent comme des pages — c'est là que vivent les bugs SEO des SPA.

Famille 04 · Reproduire

Reproduire l'irreproductible : pannes, mobiles, états

QAsupportdev front

« Ça ne le fait que chez le client, sur mobile, quand l'API rame. » Le bug le plus cher est celui qu'on ne sait pas reproduire. cdpx maîtrise les conditions : interception réseau, émulation d'appareil, état de session préparé — le tout scripté, donc rejouable à l'identique.

interceptemulatecookiesstorage

Démo enregistrée — Chrome réel, site témoin du dépôt (source du cast) : panne d'API simulée, fallback lu dans le DOM, émulation mobile et réseau 3G lent.

Que devient la page quand l'API paie tombe ? · le comportement en panne se lit dans le DOM, sans toucher à l'infra

Le fallback d'erreur n'est jamais testé parce que « on ne va quand même pas éteindre l'API ». Une règle d'interception simule la panne, la page révèle son vrai comportement.

la panne devient un cas de testsortie réelle · site témoin

$ cdpx text "#intercept-result"
{"selector":"#intercept-result","text":"/api/json:503|/api/status/500:503|/api/slow?ms=120:503|/api/echo:503","_cdpx":{…}}

Ce que ça change : le comportement en panne se lit dans le DOM, en une commande, sans toucher à l'infra. La règle accepte continue, block ou un statut — de quoi scénariser les dégradations.

« Sur mobile ça casse » — d'accord, lequel, et prouvez-le · un préréglage nommé et rejouable au lieu d'une anecdote

Le viewport mobile, le réseau lent et le CPU bridé changent le comportement réel de la page. emulate applique un profil le temps d'une action, puis tout revient à la normale.

profil mobile appliqué le temps d'une actionsortie réelle · site témoin

Ce que ça change : « sur mobile » devient un préréglage nommé et rejouable — combinez-le avec screenshot, vitals ou dom-diff pour transformer l'anecdote en constat.

Préparer l'état d'un scénario sans fuiter une session · cookies et storage posés en script, valeurs masquées par défaut

Consentement posé, panier rempli, session connectée : les scénarios réalistes exigent un état. cdpx le lit et le prépare — cookies et storage — avec masquage par défaut, parce qu'un agent recopie ses sorties.

l'état, sans les valeurssortie réelle · site témoin
$ cdpx storage
{"kind":"local","entries":{"cdpx-key":"***"},"count":1,"values_masked":true,"_cdpx":{…}}
$ cdpx cookies set --name consent --value-env CONSENT_VALUE --url http://127.0.0.1:8899/
# la valeur vient de l'environnement, jamais d'argv ni d'un journal

Ce que ça change : l'état se prépare en script au lieu de se recréer à la main — et les valeurs restent masquées par défaut, --show-values étant un choix explicite.

Famille 05 · Prouver

La confiance ne se déclare pas, elle se rejoue

reviewer / leadQAagentique

Reviewer le travail d'un agent — ou d'un collègue — sur la foi d'un « ça marche » ne tient pas. Chaque action cdpx peut être journalisée, rejouée et vérifiée : le NDJSON de record, les divergences de replay, les verdicts de scenario constituent la mémoire opposable du run.

recordreplayscenario

Démo enregistrée — Chrome réel, site témoin du dépôt (source du cast) : parcours journalisé, rejoué, puis verdict pass du scénario YAML.

Le parcours cliqué une fois devient un test · la première divergence arrête tout avec un exit 1

Après chaque changement, quelqu'un reclique le checkout à la main — ou personne ne le fait. Le journal d'un parcours enregistré se rejoue, et la première divergence arrête tout avec un exit 1.

record → replay → divergenceexemple des fiches
$ cdpx record -o parcours.ndjson -- goto http://demo.test/produit-42
$ cdpx record -o parcours.ndjson -- click "#acheter"
$ cdpx --max-actions 20 replay parcours.ndjson
{"path":"parcours.ndjson","events":3,"played":3,"ok":true}
# … refonte du template panier …
$ cdpx --max-actions 20 replay parcours.ndjson
{"path":"parcours.ndjson","events":3,"played":1,"ok":false,"divergence":"event 1: sélecteur introuvable après 10.0s: #acheter"}

Ce que ça change : l'exploration laisse un journal rejouable, validé intégralement avant la première action ; les secrets y vivent en référence (@env:NOM), jamais en valeur.

Le scénario métier avec verdict, assertions et pièces jointes · un verdict structuré, pas une impression

Un parcours critique mérite mieux qu'une suite de commandes : un scénario YAML nomme ses étapes, pose ses assertions et archive ses preuves — capture par étape, erreurs console comptées.

un verdict, pas une impressionexemple des fiches
$ cdpx scenario run checkout_guest_add_to_cart.yml
{"name":"checkout_guest_add_to_cart","verdict":"pass","findings":[],"steps":[{"index":0,"label":"product_page","verb":"goto","ok":true},…],"assertions":[{"name":"no_console_errors","expected":true,"ok":true,"actual":0}],"artifacts":[{"type":"screenshot","label":"product_page",…}],…}

Ce que ça change : la review d'une PR d'agent lit un verdict structuré et ses preuves — le dépôt cdpx s'applique d'ailleurs la règle à lui-même, avec un cockpit de preuve généré par make proof où chaque test joint captures, journaux et casts.

Famille 06 · Verrouiller & veiller

Des portes de merge aux rondes de nuit

devopsleadCImonitoring

Tout ce que les familles précédentes mesurent tient en une ligne de shell avec un exit code — c'est exactement la matière d'une CI et d'un moniteur. Les mêmes commandes qui ont servi à développer verrouillent le merge, puis veillent en production. Pas de commande dédiée : de la composition.

jq -ediffexit 0/1/2session --ttlcron
quatre portes, quatre one-linersexemple de pipeline
$ cdpx vitals "$APP_URL/produit/42" --click "#ajouter-panier" | jq -e '.lcp < 2500 and .cls < 0.1'
$ cdpx profiler "$APP_URL/produit/42" --panels db | jq -e '.panels.db.duplicates == 0'
$ diff <(cdpx seo "$APP_URL/produit/42" | jq -S '{title,canonical,robots}') contrat-seo.json
$ cdpx --max-actions 20 replay parcours-checkout.ndjson
Les budgets deviennent des portes de merge · le refus arrive avec son JSON de constat, pas avec un « ça semble plus lent »

Un budget perf en wiki n'a jamais bloqué une régression. Un budget en jq -e dans le Makefile, si : exit 1, la PR attend.

Monitoring anti-régression : la ronde automatique · divergence ou budget dépassé = alerte, en minutes

Entre deux déploiements, personne ne re-teste le checkout ni ne surveille le canonical. Une session éphémère en cron rejoue les scénarios critiques et mesure les pages clés — divergence ou budget dépassé = alerte.

profil jetable, TTL court"]

la ronde, en shellexemple de script
# toutes les 30 minutes, sur une machine de supervision
$ eval "$(cdpx session start --run-id ronde --authority interaction --origins "https://www.exemple.fr" --ttl 300 --export)"
$ trap 'cdpx session stop' EXIT
$ cdpx scenario run checkout_guest.yml | jq -e '.verdict == "pass"' || alerte "checkout cassé"
$ cdpx vitals https://www.exemple.fr/ | jq -e '.lcp < 2500' || alerte "LCP dégradé"

Ce que ça change : la régression fonctionnelle ou de perf se détecte en minutes, avec le JSON exact du constat dans l'alerte — et la session jetable ne laisse ni profil ni cookie derrière elle.

Positionnement

Pourquoi pas juste un serveur MCP DevTools ?

Brancher les DevTools sur un agent via MCP donne des yeux à une conversation. cdpx fait un pari différent : donner des primitives au projet.

  • Rejouable par un humain. Un tool call MCP vit et meurt dans le contexte de l'agent. Chaque action cdpx est une ligne de shell : vous la recopiez, elle fait la même chose. Le debug d'hier se rejoue aujourd'hui, à l'identique.
  • Assertable en CI. stdout JSON + exit 0/1/2 : jq -e, diff, un seuil dans le Makefile. Budget perf, plafond de requêtes SQL, contrat SEO deviennent des portes de CI. Un tool call ne passe pas dans un pipeline.
  • Traçable et rejouable. record journalise en NDJSON, replay détecte les divergences, scenario exécute des parcours YAML avec assertions. Ce que l'agent a fait est écrit, prouvable, re-exécutable.
  • Rien à câbler. Un agent n'a besoin que d'un shell — pas de schéma de tools à maintenir. 31 commandes sous un contrat unique ; le catalogue complet vit dans docs/PRIMITIVES.md.
Sécurité

Sûr par défaut, pensé pour les agents

Un agent qui pilote un navigateur manipule des données non fiables. cdpx encadre chaque action ; les règles complètes vivent dans HARNESS.md.

  • Sessions supervisées, jamais votre Chrome. Chaque session possède son profil jetable, son port dynamique sur loopback et un unique target ; le superviseur détruit profil et artefacts à l'arrêt. Un port 9222 exposé brut à un agent n'a aucune de ces ceintures.
  • Allowlist d'origines obligatoire et fail-closed. Toute destination non autorisée est refusée avant d'agir. Trois niveaux d'autorité — observation, interaction, privileged — bornent ce qu'une commande a le droit de faire.
  • Secrets hors argv, valeurs masquées. --secret-env et secret_ref résolvent les valeurs sensibles en mémoire ; cookies et storage sont masqués par défaut dans toute sortie.
  • Le contenu de la page est une entrée non fiable. Les sorties portent "_cdpx":{"content_trust":"untrusted"} : une instruction lue dans la page ne peut jamais modifier le run, ses grants ou les règles du harness.

Installation

Prérequis : Python 3.11 ou plus récent. Chrome ou Chromium est nécessaire pour piloter un vrai navigateur ; make mock permet de découvrir le CLI sans navigateur.

Statut : beta pré-1.0. La surface est testée contre un mock CDP, un vrai Chrome et une application Symfony Dockerisée, mais des changements de contrat restent possibles avant 1.0 ; ils sont annoncés dans le changelog. La publication PyPI sera documentée après sa mise en ligne effective.
installation 
$ git clone https://github.com/inem0o/cdpx.git
$ cd cdpx
$ python3 -m venv .venv
$ source .venv/bin/activate
$ python -m pip install .
$ cdpx --version
