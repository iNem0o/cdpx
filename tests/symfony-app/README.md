# Symfony scenario engine

This fixture app is the deterministic Symfony scenario engine used by cdpx e2e
proof runs.

## Naming convention

- Routes use `/scenario/{family}/{case}`.
- Route names use `scenario_{family}_{case}`.
- Controller methods use `{family}{Case}()` when a method owns one case, or
  `__invoke(string $case)` when a family is table-driven.
- Datasets and expected outcomes live in `App\Scenario\ScenarioCatalog`.
- Dataset ids use `{family}.{case}`, for example `profiler.baseline`.
- Artifacts generated by pytest use `{feature}.{scenario}.json` filenames and
  are attached to the `symfony` proof suite.

## Scenario families

- `profiler`: baseline/degraded, Doctrine-like query counts, N+1 duplicate
  counts, cache hit/miss and controlled payload/time signals.
- `vitals`: baseline/degraded pages with observable LCP/CLS/INP-adjacent browser
  signals.
- `rgaa`: deterministic front checks for headings, landmarks, form labels,
  focus visibility and contrast-oriented metadata. These checks represent a
  small automated RGAA subset, not full RGAA coverage.
- `front`: explicit UI states for before/after DOM regression checks.

Docker must stay self-contained. Do not add network services unless a scenario
cannot be expressed deterministically without them.
